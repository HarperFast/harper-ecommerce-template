## Workarounds Required

Three workarounds were needed to get the component running on the cluster:

1. **RocksDB lock conflict:** `next build` spawns multiple workers that each try to open the same RocksDB database. Built locally with `NODE_OPTIONS="--max-old-space-size=4096" npx next build --experimental-build-mode compile` and deployed with `prebuilt: true`.

2. **Sandbox containment:** `componentLoader.ts` loads `extensionModule` via `scopedImport` with `applicationScope`, forcing it through the VM sandbox. When `@harperdb/nextjs` dynamically imports `next`, CJS modules are loaded as ESM and fail with `exports is not defined`. Set `dependencyContainment: false` in Harper config on all cluster nodes.

## Bugs Found During Testing

### 1. Harper core: VM sandbox breaks CJS module loading for extensions (Open)

When `componentLoader.ts` loads an `extensionModule` via `scopedImport` with `applicationScope`, the extension and all its dependencies enter the VM sandbox. CJS modules like `next/dist/server/next.js` are loaded as `SourceTextModule` (ESM) where `exports` is not defined.

Error: `ReferenceError: exports is not defined`

Workaround: Set `dependencyContainment: false` in harper config.

### 2. @harperdb/nextjs: logger.loggerWithTag is not a function (Fixed in 1.2.2)

Extension called `logger.loggerWithTag()` but the correct API is `logger.withTag()`. 
Fix: https://github.com/HarperFast/nextjs/pull/35

### 3. RocksDB: multiple process lock conflict during Next.js build (Open)

`next build` spawns multiple workers that each import `harperdb` and try to open RocksDB. Only one process can hold the lock. This prevents both in-harper builds and standalone `npx next build`.

Workaround: Build with `--experimental-build-mode compile` and deploy with `prebuilt: true`.

GH Issue: https://github.com/HarperFast/rocksdb-js/issues/490

### 4. @harperdb/nextjs: GraphQL endpoint unreachable with files: '/*' (Open)

When `@harperdb/nextjs` is configured with `files: '/*'`, the Next.js request handler intercepts all routes including `/graphql`. Next.js returns its own 404 for routes that don't exist in the app directory, preventing harper's GraphQL handler from responding.

GH Issue: https://github.com/HarperFast/nextjs/issues/36

Reproduce:
```
curl -sk -X POST "https://us-ord-1.v5-testing.akamai.stage.harperfabric.com:9926/graphql" \
  -H "Content-Type: application/json" \
  -u "lavinia@harperdb.io:Stephenhawking96.," \
  -d '{"query":"{ Product { id name category price } }"}'

Expected: JSON GraphQL response
Actual: Next.js 404 HTML page
```
